Global App State Pattern

Why Use a Global App State?

In larger applications, itâ€™s helpful to centralize shared state in a single service. This is similar in spirit to a global store like NgRx, but much lighter. Signals allow us to keep things simple and reactive.

  • Centralized control over application-wide data
  • State can be read and updated from anywhere via DI
  • Easier to debug and reason about shared behavior

This example shows a shared AppStateService that holds multiple signals, such as the current user and theme. Other components can safely access or react to this state without managing it themselves.

  • user = signal<User | null>(null)
  • theme = signal('light')
  • Setter and reset methods for each field
